# M2로 넘어가기 전에 — 옮겨야 할 seam들

M1 최종 전면 리뷰(6개 렌즈 + 적대적 검증)에서 확인된, M2 착수 전에 손봐야 할 구조적 지점들이다. 여기 적힌 것 중 코드로 막은 것은 없다 — 전부 M2 설계에 반영할 항목이다.

## 1. 스텝 예산이 작업량을 재게 하라 (최우선)

`src/shell/interpreter.ts`의 `spend()`는 **명령 하나당 한 번**만 예산을 소비한다. 명령의 개수를 셀 뿐, 명령 *안*의 작업량을 세지 않는다.

M1에서 이 뿌리로부터 네 개의 탭 프리즈가 나왔다. 세 개(글롭 백트래킹, 글롭 프론티어 폭발, `mkdir -p` O(n²))는 커밋 `57df018`에서 각 핫패스에 자체 상한을 넣어 막았다. **네 번째는 남아 있다:**

### 남은 hang — grep ReDoS (의도적으로 M2 이연)

`src/shell/coreutils/grep.ts`가 플레이어의 패턴을 `new RegExp(pattern)`으로 컴파일해 `regexp.test(line)`을 돈다. `grep '(a+)+$'`를 반복 문자 뒤 비매치로 끝나는 줄에 실행하면 V8 정규식 엔진이 지수 시간으로 백트래킹한다(실측: n=28에서 3.5초, n=32는 미반환). `regexp.test`는 원자적 호출이라 JS 안에서 중간에 끊을 수 없다.

- **도달성:** 정상 플레이로는 도달 안 함(20개 퍼즐 중 중첩 수량자 패턴을 쓰는 것이 없음). 플레이어가 직접 그런 패턴을 치고, 긴 줄을 가진 파일을 만들어야(`echo`로 긴 줄을 만들어 `>`) 걸린다.
- **왜 M1에서 안 고쳤나:** grep 패턴은 임의 ERE라서, 단일 `.test()` 호출을 시간 제한으로 끊으려면 두 가지 아키텍처 변경 중 하나가 필요하다:
  1. **`exec`를 Web Worker로 옮기고** wall-clock deadline으로 워커를 `.terminate()`한다. 이것만이 폭주하는 `.test()`를 실제로 죽이는 유일한 방법이다.
  2. 백트래킹 없는 정규식 엔진(RE2류)으로 grep을 다시 짠다.

### 왜 이것이 M2 seam인가

M2가 추가할 `for` / `while` 루프는 **한 명령 안에서 무한 반복**이 가능하다 — grep ReDoS와 정확히 같은 종류의, 명령당 한 번인 스텝 예산이 못 막는 탭 프리즈다. 그러니 M2에서 반복문 안전성을 설계할 때 grep도 함께 해결하는 것이 옳다. 권장 방향:

- **`exec`를 wall-clock deadline을 가진 Web Worker로 옮긴다.** 반복문 예산 초과, grep ReDoS, 그리고 아직 발견 못 한 다른 폭주까지 전부 하나의 타임아웃 경로(`ExecutionLimitError` → exit 130, "실행 한도 초과")로 통일된다.
- 그것이 과하면, 최소한 반복 본문·확장마다 `spend`를 태워 M2 반복문만은 예산에 걸리게 하고, grep은 줄 길이 상한 같은 완화책으로 임시로 덮는다.

`store.ts`가 메인 스레드에서 `await shell.exec`를 돌리는 구조(`src/ui/store.ts`)도 워커 이전의 전제다.

## 2. 파서 / AST 확장

현재 문법은 파이프라인 · 리스트(`;` `&&` `||`) · 리다이렉션만 다룬다. `if` / `for` / `while` / `case` 복합 명령을 위한 문법 · AST 노드 · 인터프리터 실행 모델이 필요하다. 자식 컨텍스트(`interpreter.ts`의 `runSubshell` 경로)가 이미 VFS와 예산은 공유하고 env는 복사본을 뜨는 구조라, 함수 스코프의 토대는 마련돼 있다.

## 3. 함수와 위치 매개변수

인터프리터 컨텍스트에 함수 테이블과 로컬 스코프가 필요하다. 그리고 지금 리터럴 `$1`로 남는 위치 매개변수의 실제 확장을 확장기(`expand.ts`)에 넣어야 한다. M1에서 `$1`은 의도적으로 리터럴이다.

## 4. check() 계약과 echo-solvability

L1 10개 중 9개, L2 10개 중 5개가 기대 바이트를 `echo`로 찍어 "풀 수 있다". 이는 `check()`가 `ctx.history`를 읽지 않는다는 규칙의 불가피한 대가이며, 레슨을 건너뛸 뿐 거짓을 가르치지는 않는다(수용됨). 그러나 제어문을 *가르치는* L5 퍼즐은 "특정 명령을 쳤는가"가 아니라 "결과 상태가 맞는가"로만 판정하기가 더 어렵다 — 스크립트가 만들어낸 파일시스템 상태로 판정하도록 퍼즐을 설계해야 한다. M2 퍼즐 저작 시 이 제약을 먼저 고려하라.

## 5. 코어유틸 8개 추가

M2는 `sed` `awk` `cut` `tr` `uniq` `find` `xargs` `diff`를 추가한다(플랜의 "스펙과의 의도적 차이"에 이연 기록됨). `src/shell/registry.ts`의 `KNOWN_UNIMPLEMENTED`에서 새로 구현한 명령을 지워야 한다 — 안 그러면 "이 환경에는 없는 명령" 메시지가 계속 뜬다.
